Tài liệu Cấu trúc & Kỹ thuật Dự án Beatsync

Beatsync là một ứng dụng phát nhạc trên web với độ chính xác cao, cho phép đồng bộ hóa việc phát nhạc trên nhiều thiết bị cùng lúc. Dự án được xây dựng theo kiến trúc Monorepo sử dụng Turborepo.

1. Cấu trúc Tổng quan (Monorepo)

Dự án được chia thành 3 phần chính (packages) để phân tách logic và dễ quản lý:

  • apps/client Ứng dụng Front-end: Xây dựng bằng Next.js 15 (App Router), React 19, Tailwind v4, và Shadcn/ui. Quản lý giao diện hiển thị cho người dùng.
  • apps/server Máy chủ Back-end: Xây dựng bằng Bun HTTP + WebSocket server (sử dụng native Bun.serve). Quản lý phòng, đồng bộ người dùng và xử lý logic kết nối.
  • packages/shared Code dùng chung: Chứa các cấu trúc dữ liệu, chuẩn (Zod schemas) dùng chung để xác thực dữ liệu API/WebSocket giúp client và server hiểu nhau.

2. Phân tích Chi tiết Front-end (Client)

Vị trí: apps/client/

Công nghệ: Next.js, React, Zustand (Quản lý state), TailwindCSS.

Quản lý Trạng thái (State Management)

Client không lưu lộn xộn mà gom thành 3 kho lưu trữ trạng thái (store) chính bằng Zustand nằm trong apps/client/src/store/:

  • global.tsx: Quản lý các cấu hình lớn như kết nối WebSocket, bài hát đang phát, danh sách nhạc, cấu hình âm thanh không gian.
  • room.tsx: Quản lý riêng thông tin hiển thị của phòng hiện tại (mã phòng, tên người dùng).
  • chat.tsx: Xử lý hiển thị và lưu lịch sử các tin nhắn trong phòng.

3. Phân tích Chi tiết Back-end (Server)

Vị trí: apps/server/

Công nghệ: Bun, WebSocket.

Kiến trúc Server (Không dùng Database)

Điểm đặc biệt là server không cần database như MySQL hay MongoDB, mọi thứ lưu trên RAM và chia cho các người quản lý (Manager):

  • GlobalManager (Người quản lý tổng): Nắm danh sách toàn bộ các phòng đang mở.
  • RoomManager (Người quản lý phòng): Mỗi phòng có 1 quản lý này để kiểm soát người vào/ra, bài đang phát, chat.
  • BackupManager (Quản lý sao lưu): Mỗi 60 giây sẽ tự động lưu dữ liệu lên đám mây (Cloudflare R2) phòng trường hợp sập server.
  • MusicProviderManager: Quản lý việc tìm kiếm và lấy nhạc từ các nguồn bên ngoài.

4. Các Cơ chế Hoạt động Đặc biệt

4.1. Giao tiếp thời gian thực (WebSocket)

Thay vì load lại trang, mọi thao tác như nhắn tin, đổi bài hát đều truyền qua WebSocket:

  1. Tin nhắn gửi lên luôn được kiểm tra định dạng qua `WSRequestSchema` để tránh lỗi.
  2. Server phản hồi bằng 3 cách: Báo cho tất cả mọi người trong phòng (`WSBroadcast`), chỉ báo cho 1 người (`WSUnicast`), hoặc kết hợp cả 2 (`WSResponse`).

4.2. Đồng bộ hóa Phát nhạc (Cực kỳ chính xác)

Làm sao để 10 cái điện thoại phát nhạc cùng 1 mili-giây mà không bị vang/lệch tiếng?

  • Server và Client liên tục gửi tín hiệu "ping" để tính ra độ trễ mạng của từng thiết bị.
  • Khi một người bấm Play, máy chủ không ra lệnh phát ngay. Thay vào đó, máy chủ tính toán độ trễ lớn nhất và ra lệnh: "Tất cả hãy chờ sẵn, đúng 3 giây nữa đồng loạt phát nhạc nhé".
  • Các máy trễ mạng nhiều sẽ nhận lệnh muộn nhưng vẫn căn đúng giờ hẹn, các máy mạng nhanh nhận lệnh sớm thì nằm chờ. Kết quả là tất cả phát cùng lúc.

4.3. Quản lý Upload File Âm thanh

Để server không bị quá tải khi nhiều người up nhạc:

  1. Client gửi yêu cầu "Cho tôi xin cái thẻ để up file lên Cloud".
  2. Server tạo 1 "vé" tạm thời (Presigned URL) và đưa cho Client.
  3. Client dùng "vé" đó đẩy file thẳng lên máy chủ Cloud (Cloudflare R2), server không tốn băng thông trung chuyển.

5. Cách Khởi chạy Dự án Dành cho Dev

# 1. Cài đặt toàn bộ thư viện (chạy ở thư mục gốc có file package.json)

bun install

# 2. Chạy môi trường phát triển (sẽ mở cả Client và Server cùng lúc)

bun dev

# 3. Hoặc chạy từng thành phần riêng lẻ

bun client # Chỉ chạy giao diện (Cổng 3000)
bun server # Chỉ chạy máy chủ (Cổng 8080)
