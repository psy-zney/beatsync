Luồng Hoạt Động (Workflow) Hệ Thống Beatsync

1. Tổng quan Flow Hoạt động của Repository

Beatsync là hệ thống Real-time Audio Sync, được chia làm 3 trục chính giao tiếp với nhau:

  • Next.js Client (Giao diện người dùng): Gửi lệnh, phát nhạc, đồng bộ thời gian (NTP Protocol).
  • Bun Server (Xử lý trung tâm): Quản lý Room State (RAM), xử lý WebSocket, làm cầu nối xử lý các logic nặng.
  • Cloud Storage (R2/MinIO): Lưu trữ các file nhạc cache hoặc file người dùng upload để tăng tốc độ truyền tải và giảm tải băng thông.

Quy trình tiêu chuẩn của một sự kiện trong phòng:

1
Client Action: Người dùng thao tác (Play/Pause/Thêm nhạc). Client gửi Event qua WebSocket dưới dạng JSON đã được zod schema chuẩn hoá trong packages/shared.
2
Server Handling: Bun Server nhận Event, cập nhật State của Room trên RAM thông qua GlobalManager và RoomManager.
3
Broadcast: Server phát lại (Broadcast) sự thay đổi State cho toàn bộ Client trong Room cùng biết để đồng bộ giao diện.

2. Phân tích chi tiết: Tính năng Cập nhật nhạc qua Link YouTube (ytb)

Tính năng này cho phép người dùng dán 1 link YouTube vào ứng dụng và phát nhạc trực tiếp cho cả phòng nghe. Dưới đây là bảng phân tích toàn diện về tính năng này:

Khía cạnh Chi tiết Phân tích
Cách hoạt động (Mô hình Proxy & Cache) Thay vì chỉ lấy link gốc (stream URL) của YouTube và bắt Client tự load (điều này thường lỗi do CORS và URL hết hạn), hệ thống áp dụng cơ chế Phản hồi ngay - Cache ngầm (Lazy-Caching):

1. Server phân tích lấy Stream URL thực của YouTube.
2. Cung cấp ngay một URL Proxy /youtube/proxy cho Client để phát nhạc lập tức (Server sẽ proxy dữ liệu stream từ YT về cho Client).
3. Ở dưới nền, Server tải file nhạc đó xuống và upload lên S3/R2.
4. Khi tải xong, Server cập nhật lại link nhạc từ Proxy thành link S3/R2 vĩnh viễn để các lần sau không bị phụ thuộc YouTube.
Luồng di chuyển (Data Flow)
  1. Client: Gửi HTTP POST request lên /api/youtube/upload chứa roomId và url YouTube.
  2. Server (API): Gọi hàm resolveYoutubeSource(url). Kích hoạt một Child Process chạy tool yt-rust-extractor (viết bằng Rust để tối ưu tốc độ) nhằm trích xuất Stream URL mượt nhất.
  3. Server (Check Cache): Kiểm tra trong S3/R2 xem VideoID này đã được cache trước đây chưa (VD: youtube-cache/{videoId}.webm).
  4. Server (WS Broadcast): Nếu chưa cache, Server trả ngay 1 AudioSource có dạng /youtube/proxy?videoId=... và thông báo qua WebSocket để cả phòng bắt đầu phát bài hát.
  5. Server (Background Job): Chạy ngầm hàm cacheYoutubeAudioForRoom. Hàm này tải file stream và đẩy thẳng lên bucket R2.
  6. Server (WS Update): Khi Background Job hoàn thành, gửi tín hiệu WebSocket lần 2 (SET_AUDIO_SOURCES) để thay thế Proxy URL tạm thời bằng URL chuẩn của R2.
Cấu trúc Dữ liệu (Data Structure)
  • YoutubeUploadSchema: Xác thực đầu vào gồm roomId (string) và url (string dạng URL).
  • ResolvedYoutubeSource: Struct lưu trữ trung gian gồm videoId, title và streamUrl thật của Youtube.
  • CachedYoutubeStream: Lưu trong bộ nhớ RAM tạm của Server để quản lý việc expire của Stream URL: expiresAt, streamUrl, title.
  • ActiveYoutubeCacheJobs (Map): Một cấu trúc Map lưu các Promise đang chạy nền. Giúp chặn việc xử lý tải lại cùng 1 link Youtube nếu 2 người gửi cùng lúc.
  • AudioSourceSchema (Shared): Định dạng dữ liệu đẩy về Client gồm url, title, duration...
Thuật toán & Cơ chế Tối ưu (Algorithms)
  • Thuật toán Deduplication (Chống trùng lặp): Khi đang tải 1 file youtube, url được đưa vào Map activeYoutubeCacheJobs. Bất kỳ request nào vào sau hỏi cùng VideoID sẽ nhận được Promise đang chạy thay vì spawn thêm process mới.
  • Concurrency bằng Child Process: Parsing thuật toán phức tạp của YT được đẩy sang thư viện viết bằng Rust (yt-rust-extractor). Server dùng Bun.spawn để gọi Rust binary, lấy dữ liệu qua luồng stdout, tránh block Event Loop của Node/Bun.
  • Cơ chế Range Requests Proxy: Khi proxy nhạc (handleYoutubeProxy), thuật toán đảm bảo copy các Header liên quan đến Byte Range (Content-Range, Accept-Ranges) từ YT Server trả về cho Client, giúp Client có thể Tua (Seek) nhanh chóng mà không cần load cả bài.
  • Auto-Refresh Expired Streams: Nếu quá trình proxy gặp lỗi 403 Forbidden (do URL hết hạn), hệ thống tự động catch lỗi, gọi lại getYoutubeStreamByVideoId() để làm mới Stream URL và thử tải lại ngay tức thì (Retry Mechanism).
